# Frontend Implementation Spec — George

**Audience**: an autonomous coding agent (Claude Code or otherwise) implementing the Angular frontend for the George platform. The backend is in `D:\Projects\george\george-api` (Java 25, Spring Boot 4, REST + WebSocket). The frontend is in a separate Nx monorepo with two Angular apps (`landing` for visitors, `main` for operators/admins). Stack: Angular (latest, signals + standalone components), PrimeNG, Tailwind.

**Use this doc as the source of truth for all wire contracts, app responsibilities, and component conventions.** Backend types in this doc are the authoritative shapes; frontend types must match exactly.

---

## 0. Two apps, two responsibilities

| | **landing** | **main** |
|---|---|---|
| Audience | Anonymous visitors clicking SMS/email links | Authenticated tenant operators + platform admins |
| URL | `/p/:token` (single route, deep-linked) | Full SPA: `/login`, `/dashboard`, `/sessions`, `/builder`, `/conversions`, `/leads`, `/campaigns`, `/pages`, `/gateways`, `/offerings`, `/orders` |
| Auth | None — uses `wsToken` from bootstrap | Keycloak (OAuth2 PKCE via `keycloak-angular`) |
| Bundle target | < 200 KB gzipped, mobile-first | No size budget |
| PrimeNG usage | Minimal: `Toast`, `ConfirmDialog`, `InputText`, `Button`, `Calendar` | Full: `Table`, `DataView`, `Tabs`, `Dialog`, `Stepper`, `Tree`, `AutoComplete`, etc. |
| Real-time | One WebSocket per visitor session | One WebSocket per *watched* session (operators can watch many concurrently) |

Both apps share types via an Nx lib: `libs/shared-wire` (TypeScript types matching backend exactly). Maintain this lib lock-step with backend changes.

### 1.2 REST endpoints

| Method | Path | Auth | Purpose |
|---|---|---|---|
| `GET` | `/api/v1/user-session/{token}` | None (public) | Visitor bootstrap → `UserSessionBootstrap` |
| `POST` | `/api/v1/user-session/{wsToken}/submit` | None (public, wsToken in path) | Submit form → `SubmitResponse` |
| `GET` | `/api/v1/operator/sessions` | JWT | List active sessions → `ActiveSession[]` |
| `GET` | `/api/v1/operator/sessions/{sessionId}/events` | JWT | Live buffer if active, JSONB history if not → `UserEvent[]` |
| `POST` | `/api/v1/operator/sessions/{sessionId}/commands` | JWT | Push a command → 202; 429 if rate-limited; 404 if session not connected |
| `POST` | `/api/v1/inventory/{accessId}/cancel` | JWT (admin) | Cancel a service access (deprovision + status TERMINATED) |

(Plus the standard CRUD endpoints for campaigns, pages, leads, gateways, offerings, orders. They follow `/api/v1/{resource}` REST conventions; the `main` app discovers them via OpenAPI at `/v3/api-docs`.)

### 1.3 WebSocket endpoints

| Path | Auth | Direction |
|---|---|---|
| `WS /ws/visitor?wsToken=...` | wsToken from bootstrap | Visitor sends events; visitor receives operator commands |
| `WS /ws/operator?sessionId=...&token=<jwt>` | JWT in `?token=` query param | Operator subscribes to live event stream of that session |

**Close codes** (custom codes in the 4xxx range):
- `4000` — kicked because another tab opened the same session (don't reconnect)
- `4001` — auth failed (bad/missing wsToken)
- `4002` — rate limited
- `4003` — session reached terminal state (don't reconnect)
- `4004` — session not found (operator subscribing to a nonexistent session)
- `1006`, `1011`, etc. — transient; reconnect with backoff

### 1.4 Auth specifics

**Visitor**: no auth. The `wsToken` returned by bootstrap is the credential. Use it as `?wsToken=...` query param on the visitor WS. The token is ~256 random bits, unguessable. SMS link format: `{baseUrl}/p/{outreachToken}` — the visitor app exchanges that one-time outreach token for the session-scoped wsToken via the bootstrap `GET`.

**Operator**: Keycloak JWT. For REST: standard `Authorization: Bearer <jwt>` header. For WebSocket: pass JWT as `?token=<jwt>` query param (browsers can't add headers to WS handshakes). The backend's `JwtHandshakeInterceptor` decodes and validates before accepting the connection.

**Admin** (admin-only endpoints): same JWT, with `admin` realm role required (backend enforces via `@PreAuthorize("hasRole('admin')")`).

### 2.2 Lifecycle (the only happy path)

1. Visitor lands at `/p/:token`.
2. `BootstrapService` calls `GET /api/v1/user-session/{token}` → stores `sessionId`, `wsToken`, `pageDefinition` in app state (signal).
3. `PageRendererComponent` walks `pageDefinition.root` and recursively materializes Angular components via `*ngComponentOutlet` and the registry.
4. `VisitorWsService.connect(wsToken)` opens `WS /ws/visitor?wsToken=...`. On open, sends `LOAD` event with `{ url, referrer, screen, language }` payload.
5. Heartbeat timer fires every 10s, sends `{ type: 'HEARTBEAT' }`.
6. As the visitor interacts:
   - Each input (debounced 300ms) sends `{ type: 'INPUT_CHANGE', payload: { key, value } }`
   - Buttons send `{ type: 'CLICK', payload: { key } }`
   - Focus/blur send `FOCUS`/`BLUR` with `{ key }`
   - On `visibilitychange`, send `VISIBILITY_CHANGE` with `{ hidden: boolean }`
7. As operator pushes commands, `CommandDispatcher` receives them via `incoming$` and:
   - `HIGHLIGHT_FIELD` / `FOCUS_FIELD` / `SET_VALUE` / `DISABLE_FIELD` / `ENABLE_FIELD` → looks up component by `payload.key` via `DirectoryService`, calls method on component
   - `SHOW_MODAL` → opens PrimeNG `ConfirmDialog`; on user choice, sends `MODAL_RESPONSE` with `correlationId` set to the request's `id`
   - `SHOW_TOAST` / `SHOW_ALERT` → PrimeNG `MessageService`
   - `REDIRECT` → `location.href = payload.url` (or `Router.navigateByUrl`)
   - `EXECUTE_SCRIPT` — **DO NOT** evaluate arbitrary scripts. Treat as a no-op or whitelist-only specific operations. Even if the operator is trusted, this is an XSS-shaped vector.
   - `KICK_SESSION` → switch UI to `KickedComponent`, do not reconnect
   - `BLOCK_SESSION` → switch UI to terminal/blocked screen
8. On `SUBMIT` (form submission button click):
   - `SubmitStore` collects all field values from the directory.
   - Call `POST /api/v1/user-session/{wsToken}/submit` with the data.
   - On `accepted: true`: show success state. Backend marks session COMPLETED, fires Conversion record.
   - On `accepted: false` (409): session already terminal (probably abandoned). Show "this session has ended."

### 2.4 Visitor WS service skeleton

```typescript
@Injectable({ providedIn: 'root' })
export class VisitorWsService {
  private socket: WebSocket | null = null;
  private heartbeatTimer: ReturnType<typeof setInterval> | null = null;
  private reconnectAttempts = 0;
  private wsToken: string | null = null;

  readonly incoming$ = new Subject<UserEvent>();
  readonly state = signal<'idle'|'connecting'|'open'|'closed'|'kicked'|'terminal'>('idle');

  connect(wsToken: string): void {
    this.wsToken = wsToken;
    this.state.set('connecting');
    const url = `${environment.wsBase}/ws/visitor?wsToken=${encodeURIComponent(wsToken)}`;
    this.socket = new WebSocket(url);

    this.socket.onopen = () => {
      this.state.set('open');
      this.reconnectAttempts = 0;
      this.send({ type: 'LOAD', payload: { url: location.href, referrer: document.referrer } });
      this.heartbeatTimer = setInterval(() => this.send({ type: 'HEARTBEAT' }), 10_000);
    };

    this.socket.onmessage = (e) => {
      try {
        const event: UserEvent = JSON.parse(e.data);
        this.incoming$.next(event);
      } catch { /* ignore malformed */ }
    };

    this.socket.onclose = (ev) => {
      this.cleanup();
      if (ev.code === 4000) { this.state.set('kicked'); return; }
      if (ev.code === 4003 || ev.code === 4004) { this.state.set('terminal'); return; }
      if (ev.code === 4001) { this.state.set('terminal'); return; } // bad token, no recovery
      // transient — reconnect with backoff
      if (this.reconnectAttempts >= 5) { this.state.set('closed'); return; }
      const delay = Math.min(2_000 * Math.pow(2, this.reconnectAttempts), 30_000);
      this.reconnectAttempts++;
      setTimeout(() => this.wsToken && this.connect(this.wsToken), delay);
    };
  }

  send(event: Partial<UserEvent>): void {
    if (this.socket?.readyState !== WebSocket.OPEN) return;
    const full: UserEvent = { id: crypto.randomUUID(), ...event } as UserEvent;
    this.socket.send(JSON.stringify(full));
  }

  disconnect(): void {
    this.cleanup();
    this.socket?.close();
    this.socket = null;
    this.state.set('idle');
  }

  private cleanup(): void {
    if (this.heartbeatTimer) { clearInterval(this.heartbeatTimer); this.heartbeatTimer = null; }
  }
}
```

### 2.6 Component contract — what each type renders and what it accepts in props

This is the contract the **page builder** in the main app must produce, and the **visitor renderer** must consume. Everything goes in `props` (the backend `ComponentNode.props` is `Map<String, Object>`). Validators, options, and actions also go in `props` to stay consistent with the backend's generic shape.

#### Conventions
- `key` (top-level) — operator targets components by this; required, must be unique within the page
- `props.className` — string or string[] of Tailwind / PrimeNG / custom classes
- `props.style` — inline styles `Record<string, string>`
- `props.label`, `props.placeholder`, `props.helpText` — common form fields
- `props.required` — boolean
- `props.validators` — `{ type: 'required'|'min_length'|'max_length'|'pattern'|'email'|'url'|'custom', value?: string|number, errorMessage?: string }[]`
- `props.options` — for select/radio/checkbox: `{ label: string, value: string, isDefault?: boolean }[]`
- `props.actions` — `{ trigger: 'onClick'|'onChange'|'onSubmit', type: 'api-call'|'navigate'|'show-modal'|'emit-event'|'set-variable', config?: Record<string, unknown> }[]`
- `props.visible` — boolean; if false, component renders nothing

### 2.7 Edge cases

- **Visitor opens link in two tabs**: second tab gets normal connection; first tab receives close code 4000 → `KickedComponent` shown. No reconnect attempts from the kicked tab.
- **Network drops mid-session**: WS reconnects up to 5 times with exponential backoff. Heartbeat resumes. Buffer on backend is preserved (stays in registry until session terminates).
- **Network drops permanently**: after 5 retries, show "Connection lost. Please refresh the page." Frontend doesn't try to recover.
- **Visitor refreshes the page**: new bootstrap call returns existing session (server reuses session by outreach token), wsToken may be the same or rotated — frontend re-renders, reconnects WS.
- **Visitor closes tab**: WS closes; backend sees disconnect; after 30s no heartbeat → IDLE; after 90s → ABANDONED. No frontend action needed.
- **Operator pushes command while visitor is mid-flow**: command is appended to buffer + delivered. If visitor is briefly disconnected, the command is **dropped** (no offline queue). Operator UI may want to show "delivery uncertain."
- **Form submit fails (network)**: frontend retries the POST up to 3 times with backoff. After that, show "Submission failed, please try again."
- **Submit when already terminal**: backend returns 409 with `accepted: false`; frontend shows "This form has already been submitted or expired."

### 3.2 Authentication

Use `keycloak-angular`. Configure with the same realm/clientId as the backend's `SECURITY_OAUTH2_RESOURCESERVER_JWT_ISSUER_URI`. Standard PKCE flow. Token auto-refresh via the library.

For HTTP: `KeycloakBearerInterceptor` (provided by the lib) attaches `Authorization: Bearer <jwt>` to all `/api/v1/**` requests automatically.

For WebSocket: token has to go in the URL as `?token=<jwt>`. Use `keycloakService.getToken()` to fetch the current token, then build the URL.

Roles to expect in the JWT:
- `admin` — platform admin, sees admin-only screens (offerings catalog management, gateways, lead import)
- `operator` — tenant operator, sees their own sessions, conversions, orders, campaigns
- (other custom roles per realm config)

Admin-only routes guarded by `RoleGuard('admin')`.

### 3.3 Sessions list + detail (the operator's primary screen)

**Sessions list** (`/sessions`):
- PrimeNG `Table` of `ActiveSession[]`
- Columns: sessionId (short), status badge, lastHeartbeatAt (relative), bufferSize, watchers
- Click row → navigate to `/sessions/:sessionId`
- Poll `GET /api/v1/operator/sessions` every 5s (RxJS `interval`)
- Filter chips: ACTIVE / IDLE; show all by default

**Session detail** (`/sessions/:sessionId`):
- On mount: open WS via `OperatorWsService.open(sessionId, jwt)`
- Three panes:
  - **Top**: session metadata (sessionId, status, time since heartbeat, IP if available)
  - **Left**: live event stream — list of `UserEvent`, newest at top, color-coded by `incoming`, filterable by type
  - **Right**: command panel (see 3.4)
  - **Bottom**: form values being filled (derived from `INPUT_CHANGE` events accumulated by `key`)
- On unmount: close WS

### 3.5 Command bus with correlation tracking

```typescript
@Injectable({ providedIn: 'root' })
export class CommandBus {
  private pending = new Map<string, { resolve: (e: UserEvent) => void; reject: (err: unknown) => void; timer: any }>();

  constructor(private http: HttpClient) {}

  // Hook up correlation listener to a watch's events stream (call once per watch)
  attach(events$: Observable<UserEvent>): Subscription {
    return events$.subscribe(e => {
      if (!e.correlationId) return;
      const entry = this.pending.get(e.correlationId);
      if (!entry) return;
      clearTimeout(entry.timer);
      this.pending.delete(e.correlationId);
      entry.resolve(e);
    });
  }

  // Fire-and-forget
  send(sessionId: string, command: Omit<UserEvent, 'id' | 'incoming'>): Observable<void> {
    const id = crypto.randomUUID();
    return this.http.post<void>(
      `${environment.apiBase}/operator/sessions/${sessionId}/commands`,
      { id, ...command }
    );
  }

  // Send and wait for response with matching correlationId
  async sendAndAwait(
    sessionId: string,
    command: Omit<UserEvent, 'id' | 'incoming'>,
    timeoutMs = 30_000
  ): Promise<UserEvent> {
    const id = crypto.randomUUID();
    const promise = new Promise<UserEvent>((resolve, reject) => {
      const timer = setTimeout(() => {
        this.pending.delete(id);
        reject(new Error('command timed out'));
      }, timeoutMs);
      this.pending.set(id, { resolve, reject, timer });
    });
    await firstValueFrom(this.http.post(
      `${environment.apiBase}/operator/sessions/${sessionId}/commands`,
      { id, ...command }
    ));
    return promise;
  }
}
```

## 4. Page builder fixes

The existing builder code (in the user's `main` app) has contract drift from the backend. Apply these fixes.

### 4.1 Contract mismatches with backend

Backend `ComponentNode`:
```java
class ComponentNode {
  String id;
  String key;
  String type;
  Integer displayOrder;
  Map<String, Object> props;
  List<ComponentNode> children;
}
```

The builder's current `ComponentNodeDto` adds: `displayName`, `validators`, `options`, `actions` as top-level fields. These fields do not exist on the backend. Two options:

1. **Move them into `props`** (recommended) — keep backend generic, frontend uses convention.
2. **Add to backend ComponentNode** — couples backend to specific frontend feature set.

**Pick option 1.** Update the frontend `ComponentNodeDto` to match the backend exactly. The fields that the builder cares about live inside `props`:
- `props.displayName` — for builder UX label
- `props.validators` — array of validator definitions
- `props.options` — array of select options
- `props.actions` — array of action triggers
- All other fields (label, placeholder, etc.) already in props

Builder code changes required:

```typescript
// BEFORE (current builder):
export interface ComponentNodeDto {
  id: string;
  type: string;
  displayName?: string;       // <-- remove
  displayOrder?: number;
  props: Record<string, unknown>;
  children?: ComponentNodeDto[];
  validators?: Validator[];   // <-- remove (move to props)
  options?: Option[];         // <-- remove (move to props)
  actions?: Action[];         // <-- remove (move to props)
}

// AFTER (matching backend):
export interface ComponentNodeDto {
  id: string;
  key: string;                // <-- ADD: the missing required field
  type: string;
  displayOrder?: number;
  props: Record<string, unknown>;
  children: ComponentNodeDto[];
}
```

Update all builder sites that read `node.displayName`, `node.validators`, `node.options`, `node.actions` to read from `node.props['displayName']`, etc. The settings panel UI doesn't change; the underlying field path does.

### 4.2 The `key` field — currently missing

Backend requires `key`. The builder must generate it. Convention: human-readable, kebab-case, unique within page. Suggestions:

- Auto-generate from type + counter on add: `text-input-1`, `email-input-2`
- Allow user to rename in the settings panel (top field, validation: must be unique within page)
- Show validation error if duplicate

The visitor app uses `key` to register components in `DirectoryService` and to receive operator commands targeted at it. Without `key`, operator commands can't address components.

### 4.3 Page builder data model alignment

Current `PageBuilderState`:
```typescript
{
  title: string;
  slug: string;
  description: string;
  definition: {
    schemaVersion: '2.0';     // <-- not on backend
    root: ComponentNodeDto;
    variables: {};
    globalStyles: {};
    actions: {};              // <-- not on backend
  };
}
```

Backend `PageDefinition` has only `root`, `globalStyles`, `variables`. The frontend's `schemaVersion` and top-level `actions` are extras.

**Fix**: drop `schemaVersion` and `actions` from the wire shape. Keep them as frontend-internal if you want, but strip before POSTing to backend. Or (better) just remove them — they're not used.

Page itself (the wrapper) has `title`, `slug`, `description` — these are columns on the `Page` entity, separate from `PageDefinition`. Keep them in the builder state but post them as part of the page-create body alongside the definition, not inside it.

POST shape (matching backend `PageCreateUseCase` — verify by inspecting the `PageApiController` if behavior changes):
```typescript
{
  title: string;
  slug: string;
  description: string;
  definition: PageDefinition;  // { root, globalStyles, variables }
}
```

### 4.4 Option / validator / action editing — already good

The settings panel UI for these (the `<p-select>` with type list, the inputs for value/errorMessage, etc.) is fine. Just change the binding paths from `node.validators` → `node.props['validators']`.

### 4.5 Drag-drop into nested containers

The current `onDrop()` always drops at root. For a usable builder, dropping into a Container/Form/Section should add as a child of that node. The `lib-component-preview` should declare its own `pDroppable` zone passing its node id to `addComponent(type, parentId)`. The current code structure supports this (`addComponent` accepts `parentId`); the missing piece is the child droppable zone.

## 6. Cross-cutting concerns

### 6.1 Environment config

Both apps need:
```typescript
export const environment = {
  apiBase: 'http://localhost:8080/api/v1',
  wsBase:  'ws://localhost:8080',
  keycloak: {
    url: 'http://localhost:8686',
    realm: 'george',
    clientId: 'george-main',
  },
};
```

For prod: replace with HTTPS / WSS. `wsBase` must match the backend's `api.cors.allowed-origins` setting.

### 6.2 CORS

Backend `WebSocketConfig` calls `setAllowedOrigins(allowedOrigins)`. Make sure each frontend origin (dev + prod for both apps) is in the `api.cors.allowed-origins` config. Multiple origins comma-separated.

### 6.3 Error reporting

Both apps should ship errors to a backend log endpoint or a service like Sentry. The visitor app especially — when the renderer encounters bad page definitions, you want to know.

### 6.4 Versioning

The wire types are the contract. Any backend change to `UserEvent`, `ComponentNode`, `PageDefinition`, or any DTO in section 1 requires:
1. Update `libs/shared-wire`
2. Both apps recompile against new types
3. Backend deploys before frontend if the change is additive (frontend forward-compatible)
4. Frontend deploys before backend if the change is removal/rename (backend tolerant)

Coordinate via PR titles starting with `[wire]` for clarity.

## 7. Definition of done — visitor app

A visitor app is done when:
1. Visitor receives an SMS with a link, taps it, sees the rendered page within 1.5s on a 4G phone.
2. They fill the form, hit submit, see a success page.
3. The operator (in the main app) sees their entire interaction live, can push commands, and sees the matching responses.
4. Reconnect-on-network-blip works without losing data.
5. Multi-tab kick is graceful.
6. The renderer handles all the component types in section 2.6.
7. No XSS surface from `props.html` or `EXECUTE_SCRIPT` (sanitize / disable).
8. Bundle is under 200 KB gzipped.

## 8. Definition of done — main app

A main app is done when:
1. Operator logs in via Keycloak, lands on dashboard.
2. Navigates to `/sessions`, sees live list.
3. Clicks a session, sees full live stream of events, can push every command in section 3.4.
4. SHOW_MODAL with `sendAndAwait` returns the visitor's response.
5. Builder produces valid `PageDefinition`s that the visitor app can render without error.
6. CRUD works for all resources in section 3.6.
7. Roles enforced: operator can't see admin screens, admin sees everything.
